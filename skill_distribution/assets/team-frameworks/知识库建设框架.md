# 知识库建设框架

> 团队内部共享版本。剥离了个人 vault 路径和具体内容，保留 zone 架构、命名规范、图谱化读取方法论。
> Karpathy LLM Wiki 的实践版，适用于团队搭建 AI 可读的知识库。

---

## 一、Zone 架构（10 个区）

| Zone | 名称 | 用途 | 权限 |
|------|------|------|------|
| 00-系统 | 规则与模板 | 知识库规则、AI 协作准则、字段规范、模板库 | 只通过共识修改 |
| 01-原始资料 | 源材料 | 文章、PDF、播客、会议记录 | **只读，绝不修改** |
| 02-来源摘要 | 资料摘要 | 每份来源一页，提炼要点并链接相关页面 | 可更新 |
| 03-实体 | 人/公司/产品/工具 | 项目相关的实体信息 | 持续更新 |
| 04-概念 | 方法/术语/框架 | 概念定义、不同来源的说法、争议点 | 持续更新 |
| 05-主题综述 | 长期研究方向 | 汇总多来源，形成阶段性理解 | 可更新 |
| 06-对比分析 | 工具/方法对比 | 结构化比较，沉淀选择标准 | 可更新 |
| 07-问题与待研究 | 开放问题 | 没想清楚的问题、待验证假设 | 保留研究张力 |
| 08-会话沉淀 | 高价值对话 | 某次聊天形成的结论和决策 | 写后一般不修改 |

### 系统文件
- `index.md` — 总索引，重大写入后更新
- `log.md` — 操作日志，每次写入后追加
- `首页.md` — 知识库首页入口
- `工作台.md` — 日常工作台入口

### 地图页（图谱入口）
- `03-实体/实体地图.md`
- `04-概念/概念地图.md`
- `05-主题综述/主题地图.md`
- `07-问题与待研究/开放问题地图.md`

## 二、命名规范

格式：`YYYY-MM-DD 项目或主题 类型.md`

| 页面类型 | 命名示例 |
|---------|---------|
| 来源摘要 | `2026-06-08 来源名 摘要` |
| 实体页 | 直接用实体名（如 `PaddleOCR.md`） |
| 概念页 | 稳定概念可不带日期（如 `双链.md`） |
| 主题综述 | `主题名 综述` 或 `2026-06 AI Agent 趋势.md` |
| 对比分析 | `A vs B`（如 `PaddleOCR vs Tesseract.md`） |
| 问题页 | `问题：xxxx` |
| 会话沉淀 | `YYYY-MM-DD 主题 沉淀` |

禁止使用：`总结.md`、`记录.md`、`新笔记.md`、`未命名.md`

## 三、Frontmatter 字段规范

### 通用字段（所有页面）
```yaml
---
title: 页面标题
type: source-summary | entity | concept | theme | comparison | question | conversation
status: seed | active | stable | archived
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags:
  - knowledge-base
source_notes: []
related_entities: []
related_concepts: []
related_themes: []
related_questions: []
---
```

### 各类型额外字段
| 页面类型 | 额外字段 |
|---------|---------|
| 来源摘要 | `source_type`, `source_author`, `source_url`, `source_date`, `confidence` |
| 实体页 | `entity_type` (person/company/product/tool/project), `organizations`, `people` |
| 概念页 | `concept_type` (method/term/framework/model), `opposing_views`, `common_misunderstandings` |
| 主题综述 | `scope`, `key_claims`, `open_loops` |
| 对比分析 | `comparison_targets`, `decision_context`, `recommended_choice` |
| 问题页 | `question_state` (open/researching/answered/parked), `next_action` |

## 四、图谱化读取链路

核心原则：不要全库扫描。按图追踪。

```
index.md
  → 地图页（实体地图/概念地图/主题地图）
    → 相关页面（实体/概念/主题综述）
      → 来源摘要（谁说的、什么时候说的）
        → 原始资料（只验证时查阅，不日常读）
```

常见追踪路径：
- 来源摘要 → 概念页（这个来源讲了什么概念）
- 来源摘要 → 实体页（这个来源提到了谁/什么工具）
- 概念页 → 主题综述（这个概念属于哪个大方向）
- 实体页 → 主题综述（这个实体在哪个领域活跃）
- 主题综述 → 开放问题（这个主题还有哪些没闭环的问题）

## 五、摄入流程

每次有新资料时：

1. 把原资料放入 `01-原始资料`
2. 在 `02-来源摘要` 新建摘要页
3. 提炼 3-10 条关键点
4. 补充已有实体页和概念页
5. 如有必要，更新综述页和对比页
6. 在 `index.md` 补目录
7. 在 `log.md` 追加操作记录

---

## 六、写作规则

- 一条页面只讲一个清晰对象，不要胡乱合并
- 用 wikilink `[[路径/页面名]]` 串联相关页面
- 每页至少链接 2 个其他页面 — **不允许孤儿页**
- 不确定内容标记"待验证"
- 冲突内容不要硬抹平，保留双方说法并标注来源
- 优先更新已有页面，不要盲目新建

---

## 七、AI Agent 如何接入

### 会话开始时
1. 先读 `index.md` 了解知识库概况
2. 用户提到话题 → 先搜 vault 查相关笔记
3. 有笔记 → 基于笔记回答；没笔记 → 正常回答后沉淀

### 整理新知识时
1. 判断内容属于哪个 zone
2. 从模板库选合适模板
3. 写入并补齐 frontmatter 字段
4. 更新 index.md 和 log.md
5. 同时在持久记忆中存关键词索引（方便下次会话快速定位）

### 跨会话知识共享
- 知识库是唯一知识中枢
- 各工作区的 `AGENTS.md` 是轻量入口，只记该工作区需要的规则
- 分布式 Agent 通过读取同一 vault 的笔记进行参考性学习
