# 发票自动化处理框架

> 团队内部共享版本。剥离了具体公司名/金额/路径/邮箱，保留可复用的流程、规则和踩坑经验。
> 接入时替换 `{变量}` 为实际值。

## 一、完整流程（7步）

```
邮件/网盘收集 → PDF/图片解析 → 关键信息提取 → 分类与去向判断
  → 命名与分组 → 人工复核 → 归档
```

### 1. 收集
- 来源：邮件附件（IMAP/Gmail API）、云盘、网盘
- 格式：PDF、JPG、PNG
- 工具：IMAP 库 / Gmail API / 飞书 Drive / 浏览器

### 2. 解析
| 文件类型 | 工具 | 说明 |
|----------|------|------|
| 有文本层的 PDF | PyMuPDF (fitz) | 直接提取文字，不花钱 |
| 扫描件 PDF | PyMuPDF 转图片 → Vision API | 先 `fitz.Matrix(2.0, 2.0)` 渲染再调 API |
| 图片 (jpg/png) | Vision API 直接识别 | 无需预处理 |
| 已有完整文件名的 PDF | 正则从文件名提取 | 秒级处理，不走 API |

### 3. 关键信息提取（5个字段）
| 字段 | 说明 | 注意事项 |
|------|------|---------|
| 日期 | 优先用交易日期/付款日期，不要用邮件收到时间 | - |
| 金额 | Total Paid / Amount Paid，去掉逗号 | 注意全角$和NTD等非USD货币 |
| 收款方 | 实际收到钱的一方（政府机构、供应商、酒店等） | 不要把邮件发件人当收款方 |
| 付款方 | 实际付款人/付款账户显示的主体 | 不要与客户主体混淆 |
| 费用类型 | 映射到会计分类 | 不能随便写 General |

### 4. 分类与去向

#### 分类规则
| 收款方类型 | 映射分类 | 
|-----------|---------|
| AWS / Azure / GCP / 云服务商 | Computer - Software / Computer - Hosting |
| 建材/工具/维修供应商 | Repairs and Maintenance |
| 酒店/Airbnb/住宿 | Travel Expense - Lodge |
| 航空公司 | Travel Expense - Airfare |
| 租车/Uber/出租车 | Travel Expense - Car / Transportation |
| 外卖/餐厅/餐饮 | Meals and Entertainment |
| 州务卿/注册代理 | Filing Fee / Agent Fee |
| 律师/专业服务 | Professional Fees |

#### 去向规则（铁律：所有文件必须经过识别和判断）
| 文件类型 | 去向 | 说明 |
|----------|------|------|
| 律所自开发票 | 直接归入已确认 | 不参与客户发票流程 |
| 个人消费发票 | 直接归入已确认 | 不分组 |
| USPS/跳过类 | 直接归入已确认 | 无法处理或特殊标记 |
| 普通发票 | → 进待复核流程 | 必须人工确认 |

### 5. 命名规范

```
YYYYMMDDHHMM 费用类型 收款方 付款方 $金额.扩展名
```

要点：
- 日期：12位数字（年月日时分，不要秒）
- 费用类型内部保留 ` - `（如 `Travel Expense - Lodge`）
- 字段之间：普通空格
- 不使用下划线
- 金额去掉逗号：`$1,714.13` → `$1714.13`
- 去掉无用尾巴：`Receipt`、`Invoice`、`Copy`、`(1)`、`General`

### 6. 待复核分组

- 每 **20个一组**，文件夹命名：`第01组`、`第02组`...（两位前导零）
- 优先按费用类型排序，同类放一起方便批量审核
- 每组生成一个 Excel 索引表

### 7. Excel 登记格式

| 列 | 字段 | 说明 |
|---|---|---|
| 1 | 序号 | 从1开始 |
| 2 | 状态 | 待处理 / 待复核-第NN组 / 已确认 |
| 3 | 凭证文件名 | 完整文件名 |
| 4 | 日期时间 | 12位数字 |
| 5 | 费用类型 | 会计分类 |
| 6 | 收款方 | 发票开具方 |
| 7 | 付款方 | 实际付款人 |
| 8 | 金额 | 纯数字 |
| 9 | 备注 | 异常说明 |

验证规则：写入后必须用 `load_workbook` 重新读取验证行数、金额、批次。

## 二、常见坑点

### 坑1：PDF 是扫描件，PyMuPDF 提取为空
- 原因：PDF 里嵌的是图片不是文字
- 解决：先判断提取长度，短于阈值走 Vision API

### 坑2：重命名时同名文件静默覆盖导致丢失
- 原因：生成的文件名恰好与已有文件同名
- 解决：重名前做差集检查，对比 Excel 中的文件名集合和磁盘上的文件集合

### 坑3：收款方和付款方容易搞反
- 原因：PDF 中 "Bill To" 和 "From" 的位置不固定
- 规则：收款方 = 发票发出方 = 收钱的一方；付款方 = 付钱的公司

### 坑4：第三方支付平台商家名 != 公司名
- 现象：某支付平台显示的商家名（如 "Wave Team"）和实际公司名（如 "Wave Financial Inc."）不同
- 解决：以发票 PDF 正文的公司名为准，不以支付平台的显示名为准

### 坑5：邮件附件名与重命名后的文件名完全不同
- 现象：`Inv_285752_from_Vendor.pdf` → 重命名后完全无法按原名回溯
- 解决：不能按文件名匹配，必须提取 PDF 内容通过金额确认对应关系

### 坑6：政府收据的收款方判断
- 收款方 = 政府机构（如州务卿、税务部门）
- 付款方 = 实际付款人
- 系统名/简称（如 "osos"、"Commonwealth"）需要结合 PDF 画面确认正式机构名

### 坑7：Filing Fee 的客户主体识别
- 客户主体通常藏在 filing/entity/company/account/transaction details 里
- 文档上出现州政府、Secretary of State、Commonwealth、Division of Corporations → 优先作为收款方判断依据
- 反查出的公司名不放入文件名

### 坑8：金额格式不统一
- PDF 中：`$1,714.13`
- 文件名：`$1714.13`（去掉逗号）
- 可能遇到 NTD 或其他货币，不是所有都是 USD

### 坑9：分类判断错误高
- 某些费用类型（如 Shipping-Fee vs Software）AI 自动分类准确率低
- 必须人工判断
- 高风险类别（Filing Fee、Agent Fee、Shipping、Professional Fees、Consulting Fee、Travel Expense - Lodge、Meals and Entertainment）必须重点复核

### 坑10：无金额文件混入
- 酒店确认信、预订确认信等不是真实发票
- 金额可能来自取消政策说明而不是实际消费
- 必须验证文件类型：是真实发票/收据，还是确认信/政策说明/无金额页

## 三、AI Agent 接手流程

新 Agent 接手时按此顺序：
1. 确认凭证目录和 Excel 是否存在
2. 读取 Excel 和凭证文件夹
3. 检查文件名与表格是否一一对应
4. 筛出高风险类别
5. 先跑 5-10 个样本验证
6. 只修改有明确证据支持的字段
7. 修改文件名时同步更新 Excel
8. 检查无金额文件、重复文件、杂乱附件
9. 清理临时文件

---

## 四、禁止事项

- ❌ 不要把无金额文件放进最终凭证
- ❌ 不要把邮件发件人当作收款方
- ❌ 不要把客户主体和付款人混成同一个字段
- ❌ 不要把 `General` 当作费用类型
- ❌ 不要让金额出现重复标记如 `(1)`
- ❌ 不要在没有用户确认前大规模改名
- ❌ 不要每次都新建一堆交付文件，优先在现有文件上改
